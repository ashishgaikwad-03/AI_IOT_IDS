1. System Overview

The AI-IDS Dashboard is a real-time network intrusion detection system that monitors all network traffic on the host machine, classifies each packet as either benign (normal) or malicious (attack), and displays the results on a live web dashboard. It combines Machine Learning models with heuristic rules to detect various types of cyber attacks as they happen.

2. How It Works — Step by Step

Step 1: Packet Capture (Scapy)

The backend uses Scapy's AsyncSniffer to capture every IP packet that flows through the machine's network interface. This runs in passive (read-only) mode — it only observes traffic, never modifies or injects packets. The sniffer requires Administrator privileges to access the raw network interface.

Step 2: Rolling Buffer & Feature Extraction

Each captured packet is added to a rolling buffer of the last 30 packets. This buffer provides context — instead of looking at each packet in isolation, the system examines the recent traffic pattern. From this buffer, 6 key features are extracted:

#FeatureWhat It MeasuresWhy It Matters
1packet_ratePackets per second in the bufferAttacks generate much higher traffic rates than normal browsing
2unique_dst_ipsHow many different destination IPsPort scans hit many different IPs; normal traffic goes to few
3unique_src_ipsHow many different source IPsBotnet/DDoS attacks spoof many fake source addresses
4mean_pkt_sizeAverage packet size in bytesDoS floods use tiny packets; data theft uses large packets
5std_pkt_sizeVariation in packet sizesAttacks tend to have uniform sizes; normal traffic varies
6protocol_countNumber of different protocols (TCP/UDP/ICMP)Anomalous traffic often mixes protocols unusually

Step 3: Classification (Heuristic Rules + ML Model)

The system uses a two-layer classification approach:

Layer 1 — Heuristic Rules (Primary): These are pattern-matching rules specifically designed to detect known attack signatures. They check combinations of the 6 features against known attack profiles. This is the primary detection mechanism because it provides instant, reliable results.

Layer 2 — XGBoost ML Model (Secondary): An XGBoost classifier trained on the BoT-IoT dataset provides a secondary anomaly probability. If the heuristic rules don't trigger but the ML model detects elevated anomaly probability combined with unusual traffic rates, it flags the traffic as suspicious.

Why Both? The ML model alone is too conservative — it was trained on specific dataset patterns and may miss novel attack variations. The heuristic rules catch the specific attack patterns generated by our attack scripts. Together, they provide robust coverage.

Step 4: Real-Time Broadcast (WebSocket)

Every classified packet is instantly sent to all connected dashboard clients via WebSocket. The system broadcasts approximately 3 packets per second (throttled to prevent browser overload). Each broadcast includes the source IP, destination IP, protocol, port, packet size, classification label, confidence score, and severity rating.

Step 5: Dashboard Visualization (React)

The React frontend receives each packet via WebSocket and updates all dashboard components in real-time:

  • Stats Bar: Running counts of each traffic category (Benign, DDoS, DoS, Recon, Theft)
  • Threat Severity Index: A gauge (0-100) showing current threat level based on recent attack severity
  • Network Throughput Chart: Live graph of bandwidth and packets-per-second over time
  • Attack Intelligence Panel: Breakdown of detected attack types with model confidence scores
  • Live Packet Log: Scrolling table of every classified packet with color-coded rows

3. Attack Types & Detection Methods

The IDS can detect and classify the following attack types. Each has a specific traffic pattern that the system recognizes:

3.1 DoS — SYN Flood Attack

PropertyDetails
What It IsAn attacker sends a flood of TCP SYN packets (connection requests) without completing the handshake, overwhelming the target server
Attack Scripttrigger_dos.py
Traffic PatternVery high packet rate (>60 pps), tiny packets (~40 bytes each), all TCP with SYN flag, single source IP, single destination IP
How IDS DetectsChecks rolling buffer for SYN flag count > 10 combined with packet rate > 20 pps. Also detects via high rate + small mean size + few source IPs
Dashboard LabelDoS-SYN or DoS-TCP
Severity85-90 / 100

3.2 OS / IP Sweep Scan

PropertyDetails
What It IsAn attacker scans a range of IP addresses to discover which hosts are alive on the network, typically using ICMP ping packets
Attack Scripttrigger_scan.py
Traffic PatternICMP packets sent to many different destination IPs (192.168.1.1 through 192.168.1.20), moderate rate (~10 pps), small packets (~42 bytes)
How IDS DetectsChecks if ICMP packets > 30% of buffer AND unique destination IPs > 4. Also triggers when unique destinations > 8 with few source IPs
Dashboard LabelOS SCAN
Severity60-65 / 100

3.3 DDoS — Botnet Attack

PropertyDetails
What It IsMultiple compromised machines (botnet) attack a single target simultaneously. Source IPs are spoofed (faked) to hide the attacker's identity and amplify the attack
Attack Scripttrigger_botnet.py
Traffic PatternMany different source IPs (spoofed 10.0.0.x range), large packets (~540 bytes with payload), single destination, high rate (~100 pps)
How IDS DetectsChecks if unique source IPs > 8 AND mean packet size > 300 bytes. The combination of many sources + large uniform packets = botnet signature
Dashboard LabelDDoS-BOTNET
Severity92 / 100

3.4 DDoS — UDP/TCP Flood

PropertyDetails
What It IsA volumetric attack that floods the target with massive amounts of UDP or TCP traffic to consume all available bandwidth
Traffic PatternExtremely high packet rate, predominantly one protocol (UDP or TCP), multiple spoofed source IPs
How IDS DetectsFor UDP flood: UDP packets > 70% of buffer AND rate > 50 pps. For TCP flood: unique sources > 5 AND rate > 25 pps
Dashboard LabelDDoS-UDP or DDoS-FLOOD
Severity82-85 / 100

3.5 Service Scan (Port Scan)

PropertyDetails
What It IsAn attacker probes many ports on a target to find open services (like an nmap scan), which is the first step before exploiting a vulnerability
Attack Scripttrigger_svc_scan.py
Traffic PatternTCP SYN packets to many different ports on the same target, moderate rate (~20 pps), single source
How IDS DetectsDetected via high SYN count combined with moderate packet rate. The SYN-only flag pattern (no ACK) indicates connection probing
Dashboard LabelSVC SCAN
Severity52-55 / 100

3.6 Data Exfiltration / Theft

PropertyDetails
What It IsMalware or an insider sends stolen data (files, credentials, database dumps) to an external server controlled by the attacker
Attack Scripttrigger_theft.py
Traffic PatternLarge packets (~1200+ bytes) sent at high rate to a single external IP, using TCP port 443 (HTTPS) to blend with normal traffic, steady outbound stream
How IDS DetectsDetected via combination of high rate + large mean packet size + single destination. The sustained outbound flow of large packets to one IP is the signature
Dashboard LabelDATA EXFIL
Severity95 / 100

3.7 L2 Anomaly (Mixed Protocol Attack)

PropertyDetails
What It IsAn unusual traffic pattern that doesn't match any known attack but is clearly abnormal — a mix of protocols at high rate that normal applications don't generate
Attack Scripttrigger_l2_anomaly.py
Traffic PatternMix of TCP, UDP, and ICMP packets from same source, very high rate (~125 pps), small packets, random destination ports
How IDS DetectsCaught by high packet rate (>80 pps) + small mean size (<200 bytes), or when the ML model anomaly probability > 15% combined with elevated rate
Dashboard LabelL2 ANOMALY
Severity65-72 / 100

6. Technology Stack

Backend

  • Python 3.11 — Core programming language
  • FastAPI — High-performance async web framework for REST API and WebSocket
  • Scapy — Low-level packet capture and analysis library (reads raw network interface)
  • XGBoost — Gradient-boosted decision tree ML model for classification
  • Pandas / NumPy — Data processing and feature extraction
  • Uvicorn — ASGI server that runs the FastAPI application

Frontend

  • React.js — Component-based UI framework
  • Recharts — Charting library for live throughput graphs
  • WebSocket API — Browser-native real-time communication
  • Vite — Build tool for fast development and production bundling
  • CSS — Custom dark-themed SOC (Security Operations Center) design

10. Key Design Decisions

  1. Continuous Per-Packet Analysis: Instead of batching packets into time windows, each packet is processed individually with a rolling context buffer. This eliminates gaps and provides instant detection.
  2. Heuristic-First Approach: Heuristic rules provide immediate, reliable detection for known attack patterns. The ML model serves as a backup for catching anomalies that don't match predefined rules.
  3. Single-Port Deployment: Both the API backend and the static frontend are served from port 8000, so only one tunnel/port-forward is needed for public access.
  4. Auto-Detecting URLs: The frontend automatically detects whether it's running on localhost or a tunnel and adjusts API/WebSocket URLs accordingly — no manual configuration needed.
  5. Passive Monitoring: The system only reads network traffic (read-only/passive mode). It never sends, modifies, or blocks any packets.
